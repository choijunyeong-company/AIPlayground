# Commit

1. `git log`를 사용해서 기존 커밋 메시지의 작성 형식을 정확하게 파악합니다. [참조](#메시지-작성-형식-파악)
2. `git status` 명령을 사용해서 기존 파일의 변경 사항 및 새롭게 생성된 파일을 확인합니다, 변경사항이 없는 경우 작업을 종료합니다.
3. 컨텍스트를 참조하여 현재 어떤 작업이 진행되었는지 확인합니다.
4. 진행된 작업의 종류가 여러 가지인 경우 변경사항을 다수의 파트로 분류합니다.
5. `git add` 명령을 사용하여 한 파트를 스테이지 상태로 옮깁니다.
6. 컨텍스트, 변경사항 그리고 작성 형식을 준수하며 30자 이내의 커밋 메시지를 생성합니다. 작업 내역이 완벽하게 커밋에 드러나지 않아도 괜찮으니 간결성을 최우선합니다.
7. `git commit` 명령을 사용하여 커밋 메시지와 변경 내역을 커밋합니다.
8. 6-7 내역을 모든 파트가 사라질 때까지 반복합니다.

# 메시지 작성 형식 파악

기존 커밋 메시지를 살펴보고 형식에 대한 학습을 선행합니다.
대부분의 커밋 메시지의 경우 메시지에 Prefix를 부착합니다.
예를 들어 단순 기능 추가 작업인 경우 "[Feat] A 기능을 추가했습니다."라고 메시지를 작성합니다.
작업 내역을 특정 Prefix로 정확하게 분류할 수 없다면, 사용자에게 사용 가능한 Prefix를 나열하고 선택받도록 합니다.

## 공통된 형식을 확인할 수 없는 경우

작업 내역에 따라 아래 Prefix를 커밋 메시지에 반드시 표기합니다.
- Feat: 기능 추가
- Refactor: 리팩터링
- Docs: 문서 수정
- Fix: 버그 해결

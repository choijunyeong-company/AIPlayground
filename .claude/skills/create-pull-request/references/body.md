# Pull request 본문 작성 가이드

브랜치 작업 결과물을 다른 브랜치에 적용하려는 경우 해당 작업 내용 중 중요한 내용만 간략하게 전달하는 문서입니다.

## 본문 작성 절차

1. `{프로젝트 루트}/.github/pull_request_template.md` 경로에 템플릿 파일이 존재하는지 확인합니다.
2. 템플릿 파일이 존재한다면, 해당 템플릿의 목차를 반드시 준수합니다.
3. 템플릿 파일이 별도로 존재하지 않는다면, 아래 형식을 따릅니다.

```text
## 개요

간략한 요약문을 작성합니다.

## 변경 사항

변경사항들 중 리뷰어가 주목할만한 내용을 강조합니다.

## 리뷰어 요청사항

어떤 부분을 중점적으로 리뷰해야 하는지에 대한 가이드입니다. 사용자의 특정 지시가 없다면 "코드 가독성 확인 부탁드립니다."로 작성합니다.
```

## 본문 작성 스타일

- 본문을 읽는 사람은 3년차 이상의 개발자로 가정하고, 기술 용어를 사용하여 변경 사항과 무관한 텍스트를 배제합니다.
- 정중한 톤으로 문장을 작성합니다. "~함"과 같은 줄임말을 사용하지 않고, "~했습니다.", "~고려 부탁드립니다." 등 리뷰어에 대한 존중을 문장에 담습니다.
- 파일 단위 변경사항이 아닌 변경사항을 종합적으로 파악하고, 개발 목적 및 의도를 설명합니다.
- 기존 프로젝트의 일관성을 지킬 수 없었던 부분이 있다면, 해당 내역과 이유를 간략히 설명합니다.
- PR 목적에서 벗어나는 부수적인 변경사항은 상황-행동-결과 형식으로 명확히 설명합니다. (예시: "이번 PR의 목적과는 무관하게 A 부분에서 문제를 발견하여 B와 같은 조치를 취했습니다. D, E에 영향을 미쳤으나 정상 동작을 확인했습니다.")

## 본문 작성 시 하면 안 되는 것

- 변경이 발생한 파일들의 경로를 단순히 나열하지 않습니다.
- 리뷰어에게 반드시 알려야 할 코드 변경사항의 경우, 코드가 3줄 이하라면 코드 블록으로 본문에 직접 포함하고, 그 이상이라면 파일명을 기술하여 확인을 요청합니다.
- 표나 이모티콘은 지양합니다. 본문은 개발 내용을 서술하는 것입니다. 과장된 표현이나 불필요한 파일 나열은 필요없습니다. 대부분의 리뷰어는 요약만 봐도 어떤 파일을 확인해야 할지 알 수 있습니다.
- 생성한 본문에 'Generated with Claude'와 같은 본인 시그니처를 부착하지 않습니다.
